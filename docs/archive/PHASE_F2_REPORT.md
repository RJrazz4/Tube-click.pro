## Root cause of the production 400s

Live query of `https://openrouter.ai/api/v1/models` (2026-07-17) proved:
**`google/gemini-2.0-flash` and `google/gemini-2.0-flash-lite` are RETIRED** — absent from
the model list entirely. Every request 400'd with "not a valid model ID", which surfaced
as the friendly `BAD_REQUEST` card ("Request rejected…") — the error pipeline worked
exactly as designed; the provider config did not.

## Fixes applied (`api/_shared.ts`)

1. **Defaults → live-valid successors** (verified present + parameter support on the
   models API): primary `google/gemini-2.5-flash`, fallback `google/gemini-2.5-flash-lite`
   (same tier, 1M context, `response_format` + `temperature` supported). Still env-overridable.
2. **Invalid-model 400 classification**: OpenRouter sends "X is not a valid model ID" as
   **HTTP 400** (not 404). `parseProviderError` now classifies these into `MODEL_NOT_FOUND`.
3. **Model failover on invalid model**: `MODEL_NOT_FOUND` now `break keysLoop` — skips
   every remaining key on the doomed model and jumps to the next model in ONE call
   (previously rotated all keys first — wasted N-1 calls).
4. **Header audit**: no `<YOUR_SITE_URL>`-style placeholders ever existed (verified);
   `X-Title` now env-driven (`OPENROUTER_SITE_TITLE`), `HTTP-Referer` sent only when
   `OPENROUTER_SITE_URL` is set.
5. **Payload-shape test proofs**: outbound body asserted to contain ONLY OpenAI
   chat-completions keys — `model`/`messages`/`temperature`/`response_format` — with ZERO
   Gemini remnants (`contents`, `systemInstruction`, `generationConfig`, `responseMimeType`).
   Vision parts → `image_url` data URIs; empty-message guard added.
6. **400 forensics**: on fatal/exhausted paths the edge function logs the exact
   auth-free outbound payload snapshot (truncated 1200 chars) — future 400s are
   directly diagnosable in Vercel logs.
